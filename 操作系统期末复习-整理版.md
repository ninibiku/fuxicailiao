### 算法一：进程状态转换（三态/五态模型）

**基本概念：**

```
        ┌──────────┐
        │   运行态   │
        └─────┬─────┘
    时间片用完 │ 等待事件（I/O、信号量）
        ↙     │       ↘
   ┌──────┐  │   ┌──────────┐
   │ 就绪态 │←─┘   │  阻塞态   │
   └──────┘        └──────────┘
        ↑              │
        └──事件完成──────┘
```

**三态转换规则（必考）：**

| 转换 | 触发条件 |
|------|----------|
| 就绪 → 运行 | 进程调度，分得 CPU 时间片 |
| 运行 → 就绪 | 时间片用完 / 被高优先级进程抢占 |
| 运行 → 阻塞 | 等待某事件发生（主动请求 I/O、申请信号量被拒） |
| 阻塞 → 就绪 | 等待的事件已完成（I/O 中断返回、被 V 操作唤醒） |

> ⚠️ **易错提醒**：被阻塞的进程**绝对不能**直接变成运行态！  
> 阻塞 → 就绪 → 运行，必须先回就绪队列重新排队！

### 算法二：fork() 系统调用

**核心概念：**

- **调用一次，返回两次**：
  - 父进程中返回 **子进程的 PID**（> 0）
  - 子进程中返回 **0**
- **并发异步性**：不调用 `wait()` 的情况下，父子进程输出顺序是**随机、无序交替**的。

```
fork() 前：  1 个进程
fork() 后：  父进程（返回子PID） + 子进程（返回0）
             两个进程并发执行，谁先谁后不确定
```

### 算法三：调度算法 —— 计算周转时间

> 核心公式：**周转时间 = 完成时刻 − 到达时刻**  
> 带权周转时间 = 周转时间 ÷ 服务时间

**例题：** 有 3 个进程，到达时刻和运行时间如下：

| 进程 | 到达时刻 (Arrival Time) | 运行时间 (Service Time) |
|:----:|:-----------------------:|:-----------------------:|
| P1 | 0 | 6 |
| P2 | 1 | 3 |
| P3 | 2 | 1 |

---

#### 3.1 先来先服务（FCFS）—— 非抢占式

> **思想**：纯粹按到达先后顺序服务，先来先得。

| 时刻 | 事件 |
|:----:|------|
| 0 | P1 到达，直接上 CPU，一气跑到 6 |
| 6 | P1 完成。P2（时刻1到）、P3（时刻2到）在排队。先来先得，P2 先上 |
| 9 | P2 完成（6→9 跑 3ms） |
| 10 | P3 完成（9→10 跑 1ms） |

| 进程 | 开始时刻 | 完成时刻 | 周转时间 |
|:----:|:-------:|:-------:|:-------:|
| P1 | 0 | 6 | 6 − 0 = **6** |
| P2 | 6 | 9 | 9 − 1 = **8** |
| P3 | 9 | 10 | 10 − 2 = **8** |

> 平均周转时间 = (6 + 8 + 8) / 3 ≈ **7.33**

---

#### 3.2 短作业优先（SJF）—— 非抢占式

> **思想**：当 CPU 空闲时，在**已到达**的进程中挑**运行时间最短**的。一旦开跑绝不打断。

| 时刻 | 事件 |
|:----:|------|
| 0 | 只有 P1 到达，P1 上 CPU，一口气跑到 6 |
| 6 | P1 完成。P2（时刻1到）、P3（时刻2到）都在等。比较：P3(1ms) < P2(3ms)，**挑 P3** |
| 7 | P3 完成（6→7 跑 1ms）。队列只剩 P2，P2 上 |
| 10 | P2 完成（7→10 跑 3ms） |

| 进程 | 开始时刻 | 完成时刻 | 周转时间 |
|:----:|:-------:|:-------:|:-------:|
| P1 | 0 | 6 | 6 − 0 = **6** |
| P2 | 7 | 10 | 10 − 1 = **9** |
| P3 | 6 | 7 | 7 − 2 = **5** |

> 平均周转时间 = (6 + 9 + 5) / 3 ≈ **6.67**

📝 **原文修正**："挑最智短的" → "挑最短的"

---

#### 3.3 时间片轮转（RR）—— 抢占式

> **思想**：就绪队列轮流跑，每个进程最多跑一个时间片 q。时间到就被踢回队尾。  
> **RR 关键规则**：新到达的进程**先**入就绪队列，被踢出来的老进程**排在新进程后面**！

例题假设 **时间片 q = 2**：

| 时刻 | 事件 | 就绪队列结构 |
|:----:|------|:-----------:|
| 0 | P1 到达，上 CPU | [P1] |
| 1 | P2 到达，在外面等 | — |
| 2 | P3 到达；P1 时间片用完被踢下。P2→P3→P1 入队 | [P2, P3, P1] |
| 2–4 | P2 上 CPU 跑 2ms（P2 还剩 1ms）。下马排尾 | [P3, P1, P2] |
| 4–5 | P3 上 CPU。P3 只需 1ms，**提前完成**退队 | [P1, P2] |
| 5–7 | P1 上 CPU 跑 2ms（共跑 4ms，还剩 2ms）。下马排尾 | [P2, P1] |
| 7–8 | P2 跑完最后 1ms，**完成**退队 | [P1] |
| 8–10 | P1 跑完最后 2ms，**完成** | 空 |

| 进程 | 各段执行时刻 | 开始时刻 | 完成时刻 | 周转时间 |
|:----:|:-----------:|:-------:|:-------:|:-------:|
| P1 | 0–2, 5–7, 8–10 | 0 | 10 | 10 − 0 = **10** |
| P2 | 2–4, 7–8 | 1 | 8 | 8 − 1 = **7** |
| P3 | 4–5 | 2 | 5 | 5 − 2 = **3** |

> 平均周转时间 = (10 + 7 + 3) / 3 ≈ **6.67**

### 算法四：银行家算法（死锁避免）

> **类比**：银行家 = OS，客户 = 进程，资金 = 系统资源。  
> 核心原则：只有分配资源后系统仍处于**安全状态**，才允许分配。

**例题：** 当前 Available = [3, 3, 2]

| 进程 | 最大需求 Max (A,B,C) | 已分配 Allocation (A,B,C) |
|:----:|:--------------------:|:-------------------------:|
| P0 | (5, 5, 4) | (0, 1, 1) |
| P1 | (2, 2, 2) | (1, 0, 1) |
| P2 | (6, 3, 2) | (2, 1, 1) |

**第一步：计算 Need = Max − Allocation**

| 进程 | Need (A,B,C) |
|:----:|:------------:|
| P0 | (5, 5, 4) − (0, 1, 1) = **(5, 4, 3)** |
| P1 | (2, 2, 2) − (1, 0, 1) = **(1, 2, 1)** |
| P2 | (6, 3, 2) − (2, 1, 1) = **(4, 2, 1)** |

**第二步：安全性检查推演**

初始 Work = Available = [3, 3, 2]

**第一轮：**

| 检查 | 判断 | 结果 |
|:----:|------|------|
| P0 | Need [5,4,3] ≤ Work [3,3,2]？ | ❌ 不成立，跳过 |
| P1 | Need [1,2,1] ≤ Work [3,3,2]？ | ✅ 成立！P1 可完工 |
| | 回收 P1 的 Allocation [1,0,1] | 新 Work = [3,3,2] + [1,0,1] = **[4, 3, 3]** |
| P2 | Need [4,2,1] ≤ Work [4,3,3]？ | ✅ 成立！P2 可完工 |
| | 回收 P2 的 Allocation [2,1,1] | 新 Work = [4,3,3] + [2,1,1] = **[6, 4, 4]** |

**第二轮：**

| 检查 | 判断 | 结果 |
|:----:|------|------|
| P0 | Need [5,4,3] ≤ Work [6,4,4]？ | ✅ 成立！P0 可完工 |
| | 回收 P0 的 Allocation [0,1,1] | 最终 Work = [6,4,4] + [0,1,1] = **[6, 5, 5]** |

**结论：系统处于安全状态，安全序列为 P1 → P2 → P0。**

### 算法五：进程同步 —— P/V 操作与生产者-消费者

#### 5.1 信号量 S 与 P/V 操作 ⭐

| 操作 | 算法 | 物理意义 |
|:----:|------|----------|
| **P(S)** | S = S − 1；若 S < 0 → 阻塞 | **申请资源**（消费） |
| **V(S)** | S = S + 1；若 S ≤ 0 → 唤醒一个等待进程 | **释放资源**（归还） |

| S 的值 | 含义 |
|:------:|------|
| S > 0 | S = **当前可用的资源个数** |
| S < 0 | \|S\| = **当前阻塞等待的进程个数** |

#### 5.2 生产者-消费者 —— 三种信号量

| 信号量 | 初值 | 含义 |
|:------:|:----:|------|
| `mutex` | 1 | **互斥信号量**——锁住缓冲区，保证同一时刻只有一人操作 |
| `empty` | N | **同步信号量**——缓冲区空位数（还能放几个） |
| `full` | 0 | **同步信号量**——缓冲区物品数（有几个可拿） |

> ⚠️ **关键顺序（不能颠倒）**：必须先 P(empty) 再 P(mutex)！  
> 如果先锁门再申请空位，缓冲区满时会造成死锁。

#### 5.3 生产者-消费者代码骨架

```c
// 生产者
while (1) {
    生产一个产品;
    P(empty);      // ① 申请空位（先同步）
    P(mutex);      // ② 锁缓冲区（后互斥）
    放入缓冲区;
    V(mutex);      // ③ 解锁
    V(full);       // ④ 通知"有货了"
}

// 消费者
while (1) {
    P(full);       // ① 申请取货（先同步）
    P(mutex);      // ② 锁缓冲区（后互斥）
    取出产品;
    V(mutex);      // ③ 解锁
    V(empty);      // ④ 通知"腾出空位了"
    消费产品;
}
```

#### 5.4 实际应用：电商秒杀系统

| 角色 | 对应 | 做什么 |
|------|------|--------|
| **生产者** | 前端 Web 服务器 | 接收用户下单，向消息队列塞订单 |
| **缓冲区** | 消息队列（MQ） | 临时存放订单消息 |
| **消费者** | 后台数据库写入系统 | 从消息队列取订单，写入数据库 |

| 现象 | 原因 |
|------|------|
| 网页转圈/排队 | 缓冲区满 → `empty=0` → 生产者执行 P(empty) 被阻塞 |
| 凌晨消费者休眠 | 缓冲区空 → `full=0` → 消费者执行 P(full) 被阻塞 |
| 超卖 Bug | 没加 `mutex` → 两个消费者同时取到同一商品 |

**生产者/消费者各自的 PV 操作对：**

| | 生产者 | 消费者 |
|------|--------|--------|
| 干活前 | `P(empty)` 占空位 | `P(full)` 抓产品 |
| 干活后 | `V(full)` 通知有货 | `V(empty)` 腾空位 |
| 操作时 | `P(mutex)` … `V(mutex)` | `P(mutex)` … `V(mutex)` |

### 算法六：存储器管理

#### 6.1 动态分区分配算法

| 算法 | 链表排序方式 | 策略 | 缺点 |
|------|:----------:|------|------|
| **首次适应（FF）** | 按地址从低到高 | 从头找，选第一个够大的 | — |
| **最佳适应（BF）** | 按容量从小到大 | 挑刚好能装下的最小空间 | 产生大量微小外部碎片 |
| **最坏适应（WF）** | 按容量从大到小 | 每次切最大那块 | 切完剩下的仍较大，不易产生碎片 |

#### 6.2 基本分页存储 —— 物理地址变换 ⭐

> **三步法**：给定十进制逻辑地址 A 和页面大小 L，求物理地址。

| 步骤 | 公式 | 说明 |
|:----:|------|------|
| ① 求页号 P | P = A ÷ L（取整） | 逻辑地址除以页面大小 |
| ② 求偏移量 W | W = A % L（取余） | 页内偏移 |
| ③ 求物理地址 | 物理地址 = **b × L + W** | b 是查页表得到的物理块号 |

> ⚠️ **单位统一**：页面大小如果是 1KB，要换算为 1024 字节再计算！  
> 例：A = 4080，L = 1KB = 1024  
> P = 4080 ÷ 1024 = 3，W = 4080 % 1024 = 1008

### 算法七：页面置换算法（虚拟存储器）

> 当发生缺页中断且内存物理块已满时，决定**踢谁出去**。

| 算法 | 策略 | 可行性 |
|------|------|:------:|
| **OPT（最佳置换）** | 向未来看：淘汰最晚才被访问的页 | 理论最优，**无法实现** |
| **FIFO（先进先出）** | 向时间看：踢最早进入内存的页 | 可实现 |
| **LRU（最近最久未使用）** ⭐ | 向历史看：踢最久没被碰过的页 | 可实现，最常用 |

#### 7.1 LRU 算法推演示例 ⭐

> 页面引用序列：**2, 3, 2, 1, 5, 2, 4**，分配 **3 个物理块**

| 访问序列 | 2 | 3 | 2 | 1 | 5 | 2 | 4 |
|:--------:|:-:|:-:|:-:|:-:|:-:|:-:|:-:|
| 块 1（最新） | 2 | 3 | 2 | 1 | 5 | 2 | 4 |
| 块 2 | — | 2 | 3 | 2 | 1 | 5 | 2 |
| 块 3（最老） | — | — | — | 3 | 2 | 1 | 5 |
| **命中？** | ❌ | ❌ | ✔ | ❌ | ❌ | ✔ | ❌ |

> **缺页次数 = 5，命中次数 = 2**

**关键规则**：每访问一页（包括命中），该页立刻被提到"最新"位置，沉在最底的是"最老"。

**逐步推演：**

| 访问 | 操作 | 内存（新→老） | 缺页？ |
|:----:|------|:----------:|:------:|
| 2 | 块1放2 | [2] | ❌ |
| 3 | 块1放3，2下滑 | [3, 2] | ❌ |
| 2 | **命中**！2提最新 | [2, 3] | ✔ |
| 1 | 放新页1 | [1, 2, 3] | ❌ |
| 5 | 缺页！踢最老3，放5 | [5, 1, 2] | ❌ |
| 2 | **命中**！2提最新 | [2, 5, 1] | ✔ |
| 4 | 缺页！踢最老1，放4 | [4, 2, 5] | ❌ |
