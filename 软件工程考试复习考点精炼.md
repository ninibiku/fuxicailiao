# 软件工程 — 期末考试考点精炼

---

## 一、软件与软件工程基础

### 1.1 软件的定义
> 软件 = **程序** + **数据** + **文档**（三位一体，缺一不可）

- ❌ 软件 ≠ 仅仅程序
- ✅ 软件是计算机程序、数据及相关文档的完整集合

### 1.2 软件危机
- **定义**：软件开发维护中出现的严重问题（工期拖延、成本超支、质量差、维护困难）
- **软件工程诞生的原因**：为了解决软件危机

### 1.3 软件工程定义
> 用**工程化**的原则和方法开发、维护软件的学科

- **目标**：低成本、高质量、按时交付
- **三要素**：方法、工具、**过程**（注意：不含"技术"！）

### 1.4 软件工程方法学
- 编制软件的系统方法，规定各阶段活动、产品、验收步骤与完成准则
- 包括：结构化方法学、面向对象方法学等

## 二、软件生命周期

### 2.1 三大时期
| 时期 | 包含阶段 |
|------|---------|
| **定义时期** | 可行性研究 → 需求分析 |
| **开发时期** | 概要设计 → 详细设计 → 编码 → 测试 |
| **运行维护时期** | 软件维护（最后一个阶段） |

### 2.2 各阶段核心要点

| 阶段 | 核心任务 | 输出文档 |
|------|---------|---------|
| **可行性研究** | 判定项目是否值得开展（技术+经济+社会） | 可行性报告 |
| **需求分析** | 挖掘用户需求，明确功能/性能/接口要求 | **需求规格说明书** |
| **概要设计** | 将需求转为软件整体结构，划分模块 | 概要设计文档 |
| **详细设计** | 每个模块的实现细节（PAD图应用在此阶段） | 详细设计文档 |
| **编码** | 编写程序代码 | 源代码 |
| **测试** | 发现程序错误 | 测试报告 |
| **维护** | 软件交付后持续运行维护 | 维护文档 |

> ⚠️ **易错点**：软件生命周期**第一个**阶段是**可行性研究**，**最后一个**阶段是**运行维护**

## 三、模块化设计与独立性

### 3.1 模块化设计
- 将大型系统按功能拆分为多个独立、可复用的小模块
- 降低开发、调试、维护复杂度

### 3.2 耦合（Coupling）与内聚（Cohesion）

| 指标 | 定义 | 最优方向 |
|------|------|---------|
| **耦合** | 模块**之间**相互依赖程度 | **越低越好** |
| **内聚** | 模块**内部**各元素结合紧密程度 | **越高越好** |

> 🎯 **黄金法则：低耦合、高内聚**

| 耦合等级（低→高） | 内聚等级（低→高） |
|:---:|:---:|
| 无直接耦合（最优） | 功能内聚（最优） |
| 数据耦合 | 顺序内聚 |
| 控制耦合 | 通信内聚 |
| 公共耦合 | 过程内聚 |
| 内容耦合（最差） | 偶然内聚（最差） |

### 3.3 模块设计原则
- 单入口、**单出口**原则
- 作用范围应在**控制范围之内**

## 四、结构化分析与设计工具

### 4.1 常用图形工具速查

| 缩写 | 全称 | 用途 | 阶段 |
|------|------|------|------|
| **DFD** | 数据流图 | 描述**逻辑**模型，表达数据流动 | 需求分析 |
| **PAD** | 问题分析图 | 描述程序逻辑结构 | **详细设计** |
| **IPO** | 输入-处理-输出图 | 描述模块输入输出 | 概要/详细设计 |
| **甘特图** | — | 项目**进度管控** | 项目管理 |
| **工程网络图** | — | 项目进度规划 | 项目管理 |

### 4.2 DFD 图关键规则
- 每个加工必须有**一组输入 + 一组输出**数据流
- ❌ 不绘制**控制**流程，只表达数据流动
- 顶层图（上下文图）→ 0 层图 → 1 层图……逐层细化
- ⚠️ **易错**：最高层是**顶层图**（上下文图），不是 0 层图！

### 4.3 系统流程图 vs 数据流图
- 系统流程图 → 描述**物理**模型
- 数据流图 → 描述**逻辑**模型

## 五、软件测试

### 5.1 测试 vs 调试

| | 测试（Testing） | 调试（Debugging） |
|------|------|------|
| **目的** | **发现**错误 | **定位并改正**错误 |
| **性质** | 验证过程 | 纠错过程 |
| **概念** | ❌ 两者**不同** | |

### 5.2 白盒测试 vs 黑盒测试

| | 白盒测试（结构测试） | 黑盒测试（功能测试） |
|------|------|------|
| **关注点** | 程序**内部**代码逻辑 | 程序**外部**功能 |
| **依据** | 程序内部逻辑结构 | 需求规格说明书 |
| **典型方法** | 语句覆盖、判定覆盖、路径覆盖 | **等价类划分**、边界值分析 |
| **不关注** | — | 程序内部结构 |

### 5.3 逻辑覆盖强度（弱 → 强）
> 语句覆盖 ＜ 判定覆盖 ＜ 条件覆盖 ＜ 判定/条件覆盖 ＜ 条件组合覆盖 ＜ **路径覆盖**

| 覆盖类型 | 核心要求 |
|---------|---------|
| **语句覆盖** | 执行全部代码语句 |
| **判定覆盖** | 每个判断取真/假各一次 |
| **路径覆盖** | 覆盖所有可能路径（最强） |

### 5.4 测试层次

| 层次 | 测试范围 | 方法 |
|------|---------|------|
| **单元测试** | 单个模块内部逻辑 | 白盒 |
| **集成测试** | 模块间**接口与通信** | 白盒+黑盒 |
| **系统测试** | 整个系统 | 黑盒 |
| **验收测试** | 用户验证 | 黑盒 |

> ⚠️ **易错**：集成测试主要发现**总体设计**阶段错误；验收测试不用白盒！

### 5.5 α 测试 vs β 测试

| | α 测试 | β 测试 |
|------|------|------|
| **地点** | **开发环境** | **用户真实环境** |
| **执行者** | 用户（在开发者现场） | 用户（自行使用） |

### 5.6 测试用例组成
> 标准测试用例 = **输入数据** + **预期输出结果**

### 5.7 测试核心原则
- 程序员**不适合**测试自己写的程序
- 测试目的是**发现错误**，不是证明无错误
- 测试用例依据**需求规格说明书**设计（黑盒）

## 六、软件维护

### 6.1 四大维护类型

| 类型 | 含义 | 触发条件 |
|------|------|---------|
| **改正性** | 修复 BUG | 发现程序错误 |
| **适应性** | 适配环境变化 | 软硬件/平台升级 |
| **完善性** | 增加新功能/优化 | 用户新需求 |
| **预防性** | 提前优化改造 | 预防未来故障 |

### 6.2 维护工作量占比（高→低）
> 🎯 **完善性 ＞ 适应性 ＞ 改正性 ＞ 预防性**

- ✅ 完善性维护**占比最高**
- ❌ 预防性维护占比**不是**最大

### 6.3 维护副作用
- 修改代码后引发新错误的现象 → **维护副作用**
- 修改标识符引发错误 → **编码**副作用

### 6.4 软件维护 vs 硬件维护
- ❌ 两者本质**不相同**（软件不会磨损，维护性质完全不同）

## 七、面向对象基础

### 7.1 核心概念

| 概念 | 说明 |
|------|------|
| **对象** | 面向对象的基本组成单元；是**类的实例** |
| **类** | 对象的抽象模板 |
| **消息** | 对象之间通信的方式 |

> ⚠️ **易错**：对象是类的实例（不能说"类是对象的实例"！）

### 7.2 面向对象分析
- 核心模型：**对象模型**

### 7.3 UML 常用图（至少记住 5 种）
1. **用例图**
2. **类图**（描述类与相互关系）
3. **对象图**（静态结构图）
4. **时序图**（顺序图）
5. **活动图**
6. 状态图
7. 协作图
8. 构件图
9. 部署图

- 成本估计、人员组织、进度计划管理、配置管理、质量保证、文件规范
- ❌ 不包括：纯技术培训

### 8.2 进度管理工具
| 工具 | 用途 |
|------|------|
| **甘特图** | 进度管控（最常用） |
| **工程网络图** | 进度规划 |

### 8.3 CMM 模型（能力成熟度模型）

| 级别 | 特征 |
|------|------|
| 一级 | 初始级 |
| 二级 | 可重复级 |
| 三级 | 已定义级 |
| 四级 | **量化管理**级 |
| 五级 | 优化级 |

### 8.4 成本估算模型
- Putnam 模型可用于软件成本估算

### 8.5 文档编制时机
- 用户手册从**需求分析**阶段开始编制

## 九、软件质量属性

| 属性 | 含义 |
|------|------|
| **可维护性** | 软件可被理解、修改、优化的能力 |
| **可移植性** | 软件可在不同平台运行的特性 |

## 十、编码规范

- 序言性注释：模块用途、开发历史、数据说明等（❌ 不含**单行语句功能**）
- 编码风格：应尽量减少中间变量 ✅
